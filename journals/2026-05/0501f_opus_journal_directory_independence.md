
## 5. 学び - リポジトリ構造設計の原則

```
[原則1] 独立性の明示
  異質な情報体系は別ディレクトリに分ける
  (設計成果物 vs 設計プロセス記録)

[原則2] 構造的救済
  孤児ファイルを後付けで救済するより、構造変更で解消するほうが堅牢
  (本セッション: ジャーナル独立化で孤児3件が一括解消)

[原則3] 階層深さの最小化
  深い階層は相対パスを複雑にする
  03-design/journals/2026-XX/foo.md (深さ4) → journals/2026-XX/foo.md (深さ3)
  -> 修正の手間も認知的負荷も軽減

[原則4] 命名の一貫性
  番号付き (01-, 02-, 03-) と非番号 (journals/) を使い分ける
  -> 番号: 設計成果物の系統 (時系列でなく分類)
  -> 非番号: 時系列の独立した記録 (歴史)
```

## 6. ユーザの判断の妥当性検証

ユーザは0501eの「孤児3件を後続から引用するかINDEX_2026-04.mdで時系列ナビ化」という提案に対して「これはうっかりしていました　Journalは独立したディレクトリにする」と回答した。

```
私の提案 (対症療法):     INDEX_2026-04.md で時系列ナビゲーション
ユーザの判断 (構造的解決): journals/ をルート直下に独立

両者の比較:
+------------------+--------------------+--------------------+
| 観点              | INDEX案 (私)        | 独立化案 (ユーザ)    |
+------------------+--------------------+--------------------+
| 孤児3件の解消     | 1ファイル追加で対応  | 構造変更で根本解決   |
| 認知的負荷        | INDEX探す必要あり   | ルート直下で発見容易 |
| 階層深さ          | 4階層維持           | 3階層に浅化         |
| 思想的明示度      | 補助文書扱い        | 独立した記録体系     |
| 後続セッションの理解性 | 中               | 高                  |
+------------------+--------------------+--------------------+
```

ユーザの判断のほうが**構造的に堅牢**である。

これは0430iで露呈した認知傾向「Claudeは最近のセッション文脈に支配され、根本構造を見直さない」の再発である。0501e で**ディレクトリ別の検証**まで行ったにもかかわらず、私は「INDEXファイルの追加」という浅い対症療法を提案してしまった。ユーザは即座に**構造的な解決**を指示した。

> 「これはうっかりしていました」 -- ユーザの言葉

ユーザは0430l (3章構造化) の時点で「ジャーナルは設計と並列の独立した存在」と認識していたが、当時の整理で 03-design/ 配下に入れてしまったことを「うっかり」と振り返っている。0501e の検証結果がそれを露呈させた。

**Claudeの認知パターン**: 表層の対症療法を提案する  
**ユーザの認知パターン**: 構造的な解決を選択する

これは0501b (パラダイム転換) で記述された「19世紀物理学者の役割分担」の続きである:
> 「ユーザーは実験と直観、Claudeは計算と整理。Claudeは思想を保持せず、判断はユーザ。」

本セッションでは:
- Claude: ディレクトリ別検証の計算と整理 (0501e)
- ユーザ: 検証結果から構造的解決を判断 (0501f)
